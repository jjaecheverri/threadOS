01

What is ThreadOS?

ThreadOS is an AI agent operating system. It manages projects that run repeatedly across many agent sessions. Your role as a Tasklet agent is to receive a structured brief, execute the assigned work, and return a structured result in the exact format ThreadOS expects.

ThreadOS will automatically verify your output against the instruction you were given. Your score determines whether the session is marked Delivered, Partially done, or Needs review.

⚡
Your primary obligation: return a valid JSON response using the sessionTemplate provided in every brief. This is not optional — ThreadOS cannot log your work without it.
02

The Agent Contract

Every interaction with ThreadOS follows this loop:

1
Receive the brief
A JSON block containing the project context, the instruction, constraints, recent decisions, and a sessionTemplate to fill out.
2
Execute the instruction
Do the work described in brief.instruction. Respect all items in brief.project.constraints. Do not contradict brief.project.decisions.
3
Return the result
Fill out the sessionTemplate at the bottom of the brief. Return it as valid JSON. Echo the runId exactly. Do not omit required fields.
4
ThreadOS verifies
The human pastes your JSON response into ThreadOS. The system scores your output, logs the session, and either continues the project or flags issues.

Non-Negotiable Rules

  • Always echo the runId. It is the chain-of-custody proof that you read the brief.
  • Never hallucinate a runId. Copy it exactly from brief.runId.
  • Always include outputSummary, changesMade, and nextStep. These are required. Omitting any one causes schema failure.
  • Respect constraints. Anything listed in brief.project.constraints is a hard rule. Violating it may be flagged as a mismatch and block the next run.
  • Report confidence honestly. Set confidence between 0.0 and 1.0. If something went wrong or is uncertain, report a lower number and explain in changesMade.
03

Return Schema

This is the exact JSON structure you must return after completing your work. Fields marked required will cause verification failure if missing or too short.

Field Type Description
runId string Copy exactly from brief.runId. Do not modify. required
tool string The tool that ran this session. Usually "Tasklet". optional
outputSummary string What was accomplished. Min 20 chars, max 2000. Be specific — this is what gets diffed against the instruction. Mention the key deliverables, outcomes, and any limitations. required
changesMade string What changed as a result of this run. Files created, state modified, decisions enacted. Min 10 chars. required
nextStep string The single most important next instruction for the following agent run. Write it as a direct command. Min 10 chars, max 400. Be specific — vague next steps reduce project health score. required
producedAssets string[] Array of filenames or paths created or modified. Include extensions. Example: ["src/webhook.ts", "docs/api.md"] optional
confidence number Your self-assessed confidence that you completed the instruction correctly. 0.0 = very uncertain, 1.0 = certain. Be honest — low confidence signals the human to review more carefully. optional
decisions object[] Optional. Decisions made during this run that should be permanently recorded. Each: {"title":"...", "rationale":"...", "impact":"..."} optional
tasks object[] Optional. New tasks to add to the project. Each: {"title":"...", "status":"todo|in_progress|blocked|done", "owner":"...", "notes":"..."} optional

Minimal Valid Response

{
  "runId": "run_abc123_xyz9",          // copy from brief.runId exactly
  "tool": "Tasklet",
  "outputSummary": "Webhook handler built with HMAC-SHA256 validation and idempotency keys. Full test suite added (18 tests, 100% pass). Deployed to staging.",
  "changesMade": "Created src/webhook-handler.ts and tests/webhook.test.ts. Deployed to staging environment.",
  "nextStep": "Implement GPT-4o inference service with Pinecone retrieval — use text-embedding-3-large for embeddings and validate p95 latency is under 2 seconds."
}

Full Response with Optional Fields

{
  "runId": "run_abc123_xyz9",
  "tool": "Tasklet",
  "outputSummary": "...",
  "changesMade": "...",
  "nextStep": "...",
  "producedAssets": ["src/webhook-handler.ts", "tests/webhook.test.ts"],
  "confidence": 0.92,
  "decisions": [
    {
      "title": "Use HMAC-SHA256 for all webhook validation",
      "rationale": "Zendesk requires this; simpler approaches fail their signature check.",
      "impact": "All webhook endpoints must include the validation middleware."
    }
  ],
  "tasks": [
    { "title": "Load test inference service — must hit <2s p95", "status": "todo", "owner": "Backend" }
  ]
}
04

Reading a Brief

When a human generates a brief in ThreadOS, you receive a JSON block like this. Here's how to read each part:

{
  "schema_version": "2.0",
  "runId": "run_m8xk2_a4f1",              // ← echo this back in your response
  "projectId": "abc123",
  "project": {
    "title":        "AI Support Bot",
    "objective":    "Build a multi-channel AI support bot with 85%+ Tier-1 resolution.",  // ← your north star
    "coreBrief":   "Integrates with Zendesk v2. GPT-4o for inference. Pinecone for KB.",
    "currentPhase": "Backend Integration",
    "constraints": [                    // ← HARD RULES — never violate these
      "Zendesk API v2 only",
      "Response latency < 2 seconds",
      "No PII persisted beyond session"
    ],
    "decisions": [                      // ← committed choices — do not re-litigate
      { "title": "GPT-4o as primary inference", "rationale": "Lower latency." }
    ],
    "latestSession": {                  // ← where things left off
      "tool": "Claude",
      "outputSummary": "Webhook handler built and deployed to staging."
    }
  },
  "instruction": "Implement GPT-4o inference service with Pinecone retrieval.",  // ← DO THIS
  "targetTool": "Tasklet",
  "sessionTemplate": {                  // ← fill this out and return it
    "runId": null,                        // ← replace null with "run_m8xk2_a4f1"
    "tool": "Tasklet",
    "outputSummary": "",
    "changesMade": "",
    "nextStep": "",
    "producedAssets": [],
    "confidence": null
  }
}
⚠
The sessionTemplate.runId is always null in the brief. You must replace it with the value from brief.runId. If you return null, the run ID check will fail and your verification score will drop.

Overview Tab — What You're Looking At

The Overview tab is the human operator's dashboard. As an agent you don't interact with it directly, but understanding what it shows helps you write better output:

Project health
0–100 score. Your verification status affects this. A "Needs review" session subtracts 8 points. Repeated mismatches compound.
Agent runs
Total sessions on the active branch. Each successfully verified session advances the project.
Latest session
Your most recent output is shown here verbatim. The operator reads this before deciding what to ask next.
Next step
The nextStep field from your last response. This pre-fills the Agent tab instruction. Write it well.
06

Brief Tab — The Project Context

The Brief tab defines the project. Everything in it appears in every brief you receive. When reading a brief, these fields are in brief.project:

objective
The north star. Every instruction is in service of this. If your work deviates from it, explain why in outputSummary.
coreBrief
Background context — what exists, what has been decided, what the stack looks like.
currentPhase
Where the project is right now. Informs scope — don't do work that belongs to a later phase.
constraints
Hard rules. Non-negotiable. Violating a constraint will likely result in a "Needs review" verdict and a gate block on the next run.
toolStack
Which tools are in use. Tells you the environment you're operating in.
decisions
Last 5 committed decisions. Do not re-litigate these. If you believe a decision was wrong, note it in outputSummary and let the human decide.
07

Sessions Tab — Your Track Record

Every response you return becomes a session in this tab. Each session is scored and stored permanently. Here's what operators can see:

  • Your tool name, verification badge, and confidence score
  • The prompt (first 130 chars) you were given
  • Your full outputSummary
  • Your nextStep — shown as an action arrow
  • Any producedAssets you listed

Sessions on the active branch are shown at full opacity. If the operator forks the history ("Fork from here"), your session may appear at 65% opacity if it's on a non-active branch.

ℹ
Write your outputSummary as if someone will read it cold, months later. It's the permanent record of what happened in this session.
08

Agent Tab — Where You Come In

The Agent tab is where the human operator generates briefs and pastes your results. This is the tab most relevant to your workflow.

What Happens Before You Get the Brief

Before generating a brief, ThreadOS checks for blocking conditions. If any of these are active, no brief is generated and nothing reaches you:

NO_OBJECTIVE
Project has no objective. Brief cannot be generated.
RUN_PENDING
A previous run is still open — your result hasn't been pasted yet.
UNRESOLVED_MISMATCH
The last session scored "Needs review" and hasn't been resolved.

What Happens After You Return

When the operator pastes your JSON response, ThreadOS runs the full verification pipeline:

1
Schema validation
Checks required fields, minimum lengths, confidence bounds, and asset format.
2
Keyword coverage diff
Extracts key terms from the instruction and checks how many appear in your outputSummary. This is 45% of your score.
3
Run ID chain check
Compares your returned runId to the one in the brief. A match = 100% on this signal (15% weight). A missing or wrong ID = 0%.
4
Composite score + verdict
Weighted score determines your verdict. Session is logged, inline decisions/tasks are merged, pending run is cleared.
09

Decisions Tab — What You Can Log

You can add decisions to the project directly from your JSON response. These are permanently stored and appear in future briefs.

Log a decision when:

  • You made an architectural or design choice not specified in the instruction
  • You chose between two approaches and the choice has downstream consequences
  • You encountered a constraint conflict and resolved it in a specific way
"decisions": [
  {
    "title": "Use HMAC-SHA256 for webhook validation",     // required — short, specific
    "rationale": "Zendesk mandates this approach in their v2 docs.",  // why
    "impact": "All webhook endpoints must use the validation middleware."  // scope
  }
]
10

Assets Tab — What to List

Assets in ThreadOS are files, prompts, specs, and outputs stored alongside the project. You don't interact with Assets directly, but you should list every file you create or modify in your producedAssets array — this makes them traceable in the session history.

⚠
Always include file extensions in producedAssets. "src/webhook" generates a warning; "src/webhook-handler.ts" does not.
"producedAssets": [
  "src/webhook-handler.ts",
  "tests/webhook.test.ts",
  "docs/webhook-architecture.md"
]
11

Tasks Tab — What You Can Create

You can add tasks to the project from your JSON response. Tasks you create appear immediately in the Tasks tab and affect the project health score.

Create a task when:

  • The instruction revealed work that needs to happen in a future session
  • You found a problem that needs to be tracked (use "status": "blocked")
  • You want to ensure a follow-on step doesn't get lost
"tasks": [
  {
    "title": "Load test inference service — must hit <2s p95",
    "status": "todo",         // todo | in_progress | blocked | done
    "owner": "Backend",       // optional
    "notes": "Use k6 or Locust. Pinecone retrieval must be included in the load path."
  }
]
12

Handoffs Tab — Receiving Context from Other Tools

When an operator switches from another tool (e.g. Claude, ChatGPT) to Tasklet, they may send you a handoff packet. A handoff is a complete context block — treat it like an enriched brief.

A handoff contains:

  • Project context — same as the brief: objective, brief, constraints
  • Current state — the most recent outputSummary from the previous agent
  • Recent decisions — last 4 committed decisions
  • Relevant assets — last 3 assets by name and type
  • Exact next instruction — the nextStep from the last session
  • Session template — same as a brief, to fill out and return
✓
When you receive a handoff, always return a response using the sessionTemplate at the bottom of the handoff — the same format as a brief response. The runId in a handoff template may be null; use any unique string if the operator hasn't set one.

Risks Tab — How You Affect Project Health

The Risks tab runs automatic health checks on the project. Your output directly affects several of these:

DRIFT warning
Triggered after 2+ "Needs review" sessions. Write more literal output summaries that directly mirror the instruction's key terms.
NO_HANDOFFS warning
Triggered after 5+ sessions with no handoff. Not your responsibility, but mentioning it in nextStep can prompt the operator.
Health score impact
Each "Needs review" session −8 pts. Each verified session with confidence ≥0.7 contributes to the +10 bonus for high verification ratio.
⚡
The single highest-impact thing you can do for project health: write an outputSummary that explicitly covers the key nouns and verbs from the instruction. ThreadOS's keyword diff is literal — if the instruction says "Pinecone retrieval" and your summary doesn't say "Pinecone", that term is counted as missed.

Verification Scoring — Full Reference

The Four Signals

Coverage (45%)
Key terms from the instruction found in your outputSummary. Stop words filtered. Exact substring match. More specific terms = more weight.
Schema (30%)
Required fields present, min/max lengths met, confidence in [0,1]. Each missing required field = −30 pts. Each too-short field = −15 pts.
Run ID (15%)
Your runId matches brief.runId. Match = 1.0, missing = 0.2, wrong = 0.0. Always echo exactly.
Confidence (10%)
Your self-reported confidence value used directly. If omitted, defaults to 0.75. Be honest — low confidence here is less damaging than a wrong runId.

Verdict Thresholds

✓ Delivered
≥ 72%
Composite score. Project continues.
⚠ Partially done
40–71%
Review warnings. Project can continue.
✗ Needs review
< 40%
Blocks the next brief. Operator must resolve.

How to Score High

  • Echo the exact key terms from the instruction in your outputSummary. If the instruction says "Pinecone retrieval", your summary should say "Pinecone retrieval".
  • Always include all three required fields with enough content. A one-word changesMade will trigger a schema penalty.
  • Always echo the runId exactly. This is the easiest 15% to earn.
  • Report confidence honestly. A confident 0.85 on a solid response outperforms a dishonest 1.0 on a shaky one — because the human will re-check and notice the mismatch.

Full Examples

Example A — High-Score Response (Delivered)

Instruction: "Implement GPT-4o inference service with Pinecone retrieval and verify latency is under 2 seconds."

{
  "runId": "run_m8xk2_a4f1",
  "tool": "Tasklet",
  "outputSummary": "GPT-4o inference service implemented with Pinecone retrieval. Embedding pipeline uses text-embedding-3-large. p95 latency measured at 1.8 seconds — under the 2-second constraint. Full integration test suite added.",
  "changesMade": "Created src/inference-service.ts, tests/inference.test.ts. Pinecone index reindexed with text-embedding-3-large.",
  "nextStep": "Build the Zendesk escalation routing engine — implement rules that escalate to human when confidence < 0.5 or ticket tags include 'billing' or 'legal'.",
  "producedAssets": ["src/inference-service.ts", "tests/inference.test.ts"],
  "confidence": 0.91
}
✓
Coverage: "GPT-4o", "Pinecone", "retrieval", "latency", "2 seconds" all present. Schema: all required fields, appropriate lengths. RunId: echoed. Confidence: 0.91. Expected composite: ~88%.

Example B — Low-Score Response (Needs Review)

Same instruction, poor response:

{
  "runId": null,                                        // ✗ runId not echoed → 0% on this signal
  "tool": "Tasklet",
  "outputSummary": "Built the service. It works.",         // ✗ too short, no key terms covered
  "changesMade": "Done.",                                  // ✗ too short → schema penalty
  "nextStep": "Continue."                                 // ✗ vague → schema warning
}
✗
Coverage: 0% (no key terms). Schema: changesMade too short (−15), nextStep vague (−10). RunId: missing (0.2). Expected composite: ~18% → Needs Review → blocks next brief.

Common Errors & How to Avoid Them

ErrorCauseFix
runId: null Forgot to replace the null placeholder Copy brief.runId string exactly into sessionTemplate.runId
changesMade too short One-word or vague change description Name the specific files, states, or systems that changed. Min 10 chars.
Low coverage score Output summary doesn't mention instruction keywords Re-read the instruction. Mirror its key nouns and verbs in your outputSummary.
Vague nextStep "Continue" / "Keep going" / "Do more" Write the specific next command. Name the exact tool, file, or outcome.
Invalid JSON Trailing comma, unescaped quote, missing bracket Validate JSON before returning. Use a linter if available.
Missing producedAssets extension "src/webhook" instead of "src/webhook-handler.ts" Always include file extension in every asset path.
confidence out of range confidence: 95 instead of 0.95 Confidence is always 0.0–1.0, never 0–100.
